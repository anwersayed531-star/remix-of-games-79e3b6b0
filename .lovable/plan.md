

# خطة تحسين نظام اللعب الجماعي المحلي

## ملخص المشروع
تحويل نظام الاتصال الحالي (نسخ/لصق الأكواد) إلى نظام سهل يعتمد على QR Code مع دعم بطولات تصل إلى 40 لاعب، وشريط جانبي لمتابعة المباريات.

---

## المرحلة 1: نظام QR Code للاتصال

### كيف يعمل (مثل ShareIt)
- المضيف ينشئ غرفة فيظهر QR Code على شاشته
- اللاعبين الآخرين يفتحون الكاميرا من التطبيق ويمسحون الكود
- الاتصال يتم تلقائيا بدون نسخ/لصق

### المطلوب تقنيا
- تثبيت مكتبة `qrcode.react` لعرض QR Code
- تثبيت مكتبة `html5-qrcode` لقراءة QR Code بالكاميرا
- استخدام Capacitor Camera permission للأندرويد الحقيقي
- إنشاء مكون `QRCodeDisplay` لعرض الكود
- إنشاء مكون `QRCodeScanner` لمسح الكود بالكاميرا مع طلب إذن الكاميرا
- تعديل `MultiplayerLobby` ليستخدم QR بدل النسخ/اللصق

### تدفق الاتصال الجديد
```text
+------------------+       +------------------+
|   المضيف         |       |    اللاعب        |
|                  |       |                  |
| 1. إنشاء غرفة   |       |                  |
| 2. يظهر QR Code  |  <--  | 3. يفتح الكاميرا |
|                  |       | 4. يمسح QR Code  |
| 5. اتصال تلقائي  | ----> | 5. اتصال تلقائي  |
+------------------+       +------------------+
```

## المرحلة 2: دعم اتصال متعدد (2-40 لاعب)

### المشكلة الحالية
النظام الحالي WebRTC peer-to-peer يدعم لاعبين فقط (1-to-1).

### الحل: شبكة نجمية (Star Topology)
- المضيف يكون "السيرفر المحلي" ويتصل مع كل لاعب بـ WebRTC DataChannel منفصل
- كل لاعب جديد يمسح QR الخاص بالمضيف
- المضيف يعيد توزيع الرسائل لكل اللاعبين

```text
        لاعب 1
          |
لاعب 2 -- المضيف -- لاعب 3
          |
        لاعب 4
```

### المطلوب تقنيا
- إنشاء `useP2PHost` hook: يدير اتصالات متعددة (مصفوفة من RTCPeerConnection)
- إنشاء `useP2PGuest` hook: اتصال واحد بالمضيف
- تعديل `useMultiplayerSync` ليدعم البث لكل اللاعبين
- المضيف يولّد offer جديد لكل لاعب يمسح QR (يتم تضمين offer في QR ويتم answer تلقائيا)

## المرحلة 3: نظام البطولات والمجموعات

### الفكرة
- المضيف (صاحب الغرفة) يقدر يحدد عدد اللاعبين في كل مباراة (2-4)
- المضيف يقسم اللاعبين لمجموعات يدويا أو تلقائيا
- كل مجموعة تلعب مباراتها بشكل مستقل ومتزامن
- اللي يخسر يقدر يتفرج على المباريات التانية

### الملفات الجديدة
- `src/components/TournamentManager.tsx` - واجهة إدارة البطولة
- `src/hooks/useTournament.ts` - منطق البطولة (تقسيم المجموعات، تتبع النتائج)
- `src/components/TournamentBracket.tsx` - عرض شجرة البطولة

### تدفق البطولة
```text
1. المضيف ينشئ غرفة
2. اللاعبين يمسحوا QR وينضموا (حتى 40)
3. المضيف يحدد عدد اللاعبين في كل مباراة
4. المضيف يقسم المجموعات (يدوي أو تلقائي)
5. كل مجموعة تبدأ مباراتها
6. الخاسرين يتفرجوا على المباريات الباقية
7. الفائزين يستمروا في الأدوار التالية
```

## المرحلة 4: الشريط الجانبي للمباريات

### الوصف
شريط جانبي رفيع قابل للتحريك يعرض:
- كل المباريات الجارية حاليا
- حالة كل مباراة (جارية/منتهية)
- نتائج المباريات
- إمكانية المشاهدة للمتفرجين (الخاسرين)

### خصائص الشريط
- رفيع بشكل افتراضي (48px تقريبا)
- قابل للسحب والتحريك لأي جانب (يمين/يسار/أعلى/أسفل)
- يتوسع عند الضغط ليعرض التفاصيل
- يحفظ موقعه في localStorage

### الملفات الجديدة
- `src/components/MatchSidebar.tsx` - الشريط الجانبي
- `src/hooks/useDraggableSidebar.ts` - منطق السحب والتحريك

## المرحلة 5: ضمان الاتصال المحلي بدون إنترنت

### الوضع الحالي
يستخدم STUN server من Google (يحتاج إنترنت).

### التحسين
- على نفس شبكة WiFi: WebRTC يعمل بدون STUN غالبا - سنزيل الاعتماد على STUN كأساسي
- إضافة mDNS candidates للشبكة المحلية
- إزالة timeout الاتصال (المستخدم يتحكم بالإلغاء يدويا فقط)
- إضافة إعادة محاولة تلقائية بدل عرض "فشل الاتصال"
- إبقاء STUN كـ fallback فقط
